# design-craft.md — 디자인 워크플로우 평가 기준

Pencil Design 스킬이 관여하는 **디자인 변환 워크플로우**의 품질을 평가한다.
글쓰기 5축 평가(writing-craft.md)와는 독립적인 체계이며, 디자인 산출물 고유의 기준으로 평가한다.

---

## 1. 평가 대상 케이스

| 케이스 | 흐름 | 트리거 조건 |
|--------|------|------------|
| **Case A** (Template) | WPF XAML 조사 → wpf-animation.pen 템플릿 보강 | "wpf-템플릿조사 후 템플릿보강해" 등 |
| **Case B** (Project) | wpf-animation.pen 참고 → 프로젝트 .pen 디자인 (정적+애니메이션 가이드) | "wpf-animation 이펙트를 참고해 OO 디자인해줘" 등 |
| **Case W** (Web) | .pen 파일 참조 → HTML/웹 구현 | .pen 파일을 참고하여 HTML/CSS/JS 페이지 작성 완료 시 |

Case A와 B는 독립적으로 발동할 수도, 연속 파이프라인(A → B → W)으로 발동할 수도 있다.

## 2. Case A: WPF 조사 → 템플릿 보강 (3축, 100점)

wpf-animation.pen에 **새로운 WPF 애니메이션 기법 카드를 추가**하는 워크플로우.

### 축 A1: 리서치 신규성 & 비중복성 (35점)

기존 wpf-animation.pen에 이미 있는 카드와 **중복 없이** 새로운 기법을 추가했는가.

| 점수 | 기준 |
|------|------|
| 0 | 기존 요소와 대부분 중복, 새로운 기법 없음 |
| 15 | 1-2개 신규 기법이나 기존과 유사한 변형 수준 |
| 25 | 3개+ 신규 기법, 기존 카테고리와 명확히 구분됨 |
| 35 | 기존에 없는 카테고리/패러다임 도입 + 독창적 기법 발굴 |

**판정 방법:**
```
1. 기존 wpf-animation.pen의 카테고리/카드 목록 수집 (batch_get)
2. 새로 추가한 카드와 대조
3. 중복 = 동일 WPF 속성(예: DoubleAnimation on Opacity)을 같은 컨트롤에 적용
4. 신규 = 이전에 없던 WPF 속성 조합, 새 컨트롤, 새 인터랙션 패턴
```

### 축 A2: 시각화 표현력 (35점)

정적 이미지로 **애니메이션 전후 상태(Before → After)**를 직관적으로 전달하는가.

| 점수 | 기준 |
|------|------|
| 0 | 텍스트 설명만 존재, 시각적 상태 표현 없음 |
| 15 | 단일 상태만 표현 (Before 또는 After만) |
| 25 | Before/After 양쪽 표현 + 화살표/전환 표시 |
| 35 | 다단계 상태 전환 + 색상/크기/위치 변화 시각적 차이 명확 + 레이아웃 일관성 |

**판정 방법:**
```
1. get_screenshot()으로 각 카드의 시각적 미리보기 영역 확인
2. 상태 전환 표현 패턴 식별:
   - Hover 전환: Normal → Hover 나란히 배치 여부
   - 색상 변화: 그라데이션 단계 표현 여부
   - 크기 변화: 오버레이/나란히 배치 여부
   - 회전: 호(arc) 또는 각도 표시 여부
3. 전체 카드 간 시각적 일관성 (색상 체계, 간격, 폰트)
```

### 축 A3: 기술 메타 완결성 (30점)

XAML 코드 스니펫과 조사 히스토리가 완전하고 정확한가.

| 점수 | 기준 |
|------|------|
| 0 | 코드 스니펫 없음, 출처 기록 없음 |
| 12 | 코드 스니펫 있으나 불완전하거나 출처 미기록 |
| 22 | 실행 가능한 XAML 스니펫 + research-history.md 기록 |
| 30 | 독립 실행 가능한 .xaml 샘플 파일 + 출처 URL + 핵심 기술 키워드 기록 |

**판정 방법:**
```
1. design/xaml/sample/*.xaml 파일 존재 및 WPF Window 독립 실행 가능성 확인
2. design/xaml/research-history.md에 출처 URL, 날짜, 핵심 기술 키워드 기록 여부
3. wpf-animation.pen 내 코드 블록의 XAML 문법 정확성
```

## 3. Case B: wpf-animation.pen 참고 → 프로젝트 디자인 (3축, 100점)

wpf-animation.pen의 애니메이션 기법을 **참고**하여 프로젝트 요구사항에 맞는 새로운 .pen 파일을 디자인하는 워크플로우.
**핵심 원칙: 정적 디자인(룩앤필)과 동적 정의(애니메이션 가이드)를 분리한다.**

### 축 B1: 요구사항 충실도 (35점)

사용자가 요구한 **페이지/기능/UX 흐름**이 정적 디자인에 모두 반영되었는가.

| 점수 | 기준 |
|------|------|
| 0 | 요구사항 대부분 누락, 페이지 구성 불완전 |
| 15 | 주요 페이지 존재하나 일부 기능 누락 |
| 25 | 전체 페이지 구성 완료, 주요 기능 모두 반영, UX 흐름 자연스러움 |
| 35 | 전 요구사항 커버 + 상태(로딩/에러/빈값) 표현 + 사이드바/네비 재사용 컴포넌트 활용 |

**판정 방법:**
```
1. 사용자 요구사항 목록 추출 (메뉴, 기능, 유효검사 등)
2. .pen 파일 화면별 매핑: 요구사항 → 구현된 프레임
3. 커버리지 = (구현된 요구사항 / 전체 요구사항) × 100%
4. UX 흐름: 화면 간 네비게이션 일관성, 사이드바 상태 변화
```

### 축 B2: 애니메이션 가이드 풍부성 (35점)

wpf-animation.pen의 기법을 **얼마나 다양하게 참고·매핑**하여 프로젝트에 적용했는가.
각 애니메이션 카드가 **Target 프레임 명시 + Before→After 시각화 + XAML 코드**를 갖추었는가.

| 점수 | 기준 |
|------|------|
| 0 | 애니메이션 가이드 없음 또는 텍스트 메모 수준 |
| 15 | 3개 이하 애니메이션 카드, wpf-animation.pen 참고 카테고리 1-2개 |
| 25 | 6개+ 카드, 3개+ wpf-animation.pen 카테고리 참고, Target/Before→After/XAML 구조 |
| 35 | 화면별 전담 카테고리 구성 + 8개+ 다양한 기법 + 전 카드 완결된 구조 + 룩앤필 매칭 |

**판정 방법:**
```
1. 애니메이션 가이드 프레임 수 및 카드 수 집계
2. 각 카드가 참고한 wpf-animation.pen 카테고리 식별 (CAT1~10 매핑)
3. 카드 구조 완결성: Title + Target 명시 + Description + Before→After Visual + XAML Code
4. 룩앤필 일관성: 애니메이션 가이드 테마가 정적 디자인과 매칭되는가 (중요!)
```

### 축 B3: 디자인 품질 & 분리 기법 (30점)

정적 디자인의 **시각적 완성도**와 정적↔동적 **분리 아키텍처**가 올바른가.

| 점수 | 기준 |
|------|------|
| 0 | 레이아웃 깨짐, 분리 없이 혼재 |
| 12 | 기본 레이아웃 완성, 분리 시도하나 불완전 |
| 22 | 일관된 테마/폰트/색상 + 정적·동적 명확히 분리 + 애니메이션에서 적용 위치 참조 |
| 30 | 디자인 시스템 활용 (재사용 컴포넌트) + 완벽한 분리 + 스타일 가이드 준수 + 반응형 고려 |

**판정 방법:**
```
1. 스크린샷으로 시각적 완성도 확인 (정렬, 간격, 색상 일관성)
2. 분리 확인: 정적 화면 프레임과 애니메이션 가이드 프레임이 물리적으로 분리
3. 참조 확인: 애니메이션 카드에서 "Target: {Screen}/{Frame}" 형태로 적용 위치 명시
4. 재사용: sidebar 등 공통 컴포넌트를 reusable로 정의하고 ref로 사용
5. 룩앤필 매칭: 정적 디자인이 밝은 테마면 애니메이션 가이드도 밝은 테마
```

## 4. Case W: Pencil → HTML/웹 구현 (3축, 100점)

### 축 W1: 디자인 요소 커버리지 (35점)

.pen 파일에 정의된 컴포넌트/이펙트를 HTML 구현에 **얼마나 반영**했는가.

| 점수 | 기준 |
|------|------|
| 0 | .pen 참조 없이 독자적으로 작성 (디자인 무시) |
| 15 | 1-2개 카테고리의 요소만 부분 적용 |
| 25 | 3개+ 카테고리에서 요소 적용, 주요 컴포넌트 대부분 반영 |
| 35 | 전 카테고리 커버 + 각 카테고리별 대표 요소 1개 이상 구현 |

**판정 방법:**
```
1. .pen 파일의 카테고리 및 컴포넌트 목록 수집
2. HTML/CSS/JS에서 해당 요소가 구현된 부분 매핑
3. 커버리지 = (구현된 컴포넌트 수 / 전체 컴포넌트 수) × 100%
4. 매핑 테이블 작성 (pen 요소 → HTML 구현 위치)
```

### 축 W2: 애니메이션 구현 충실도 (35점)

정적 디자인으로 표현된 애니메이션 의도를 **실제 동적 애니메이션**으로 충실히 변환했는가.

| 점수 | 기준 |
|------|------|
| 0 | 애니메이션 없는 정적 HTML |
| 15 | 기본 CSS transition만 사용 (hover 색상 변경 수준) |
| 25 | CSS animation + transition 다양하게 활용, 3종+ 애니메이션 패턴 |
| 30 | CSS + JS 조합으로 복합 애니메이션 구현 + 원본 디자인 의도 정확 반영 + 인터랙션 |
| 35 | 위 기준 + **다중 CAT 활용(4개+)** + **WAAPI 순차 시퀀스** + **SVG 리얼리티(그라데이션/통합 구조)** |

**판정 방법:**
```
1. WPF → Web 매핑 충실도 확인:
   - QuadraticEase EaseIn → JS quadratic easing (t*t)
   - SineEase EaseInOut → CSS ease-in-out 또는 cubic-bezier
   - AutoReverse + Forever → CSS alternate infinite
   - ScaleTransform → transform: scale()
   - RotateTransform → transform: rotate() + @keyframes
   - TranslateTransform → translateX/Y()
   - RepeatBehavior="Forever" → animation: infinite 또는 rAF loop
   - BeginTime stagger → animation-delay 증분
   - Path animation (stroke-dashoffset) → SVG stroke-dasharray/dashoffset
   - ElasticEase → cubic-bezier(0.175, 0.885, 0.32, 1.275)
   - 파티클 시스템 → JS Canvas rAF 또는 WAAPI
   - 글로우/펄스 → filter: drop-shadow() 또는 box-shadow animation
2. SVG 리얼리티 품질:
   - 자연물(꽃잎, 잎)에 radialGradient 적용 여부 (flat fill = 감점)
   - 복합 요소(꽃+줄기+잎)의 통합 SVG 구조 여부 (따로 놀면 감점)
   - 초기 등장 시 fade-in 처리 여부 (떨림 방지)
3. 순차 애니메이션 구현:
   - Web Animations API(WAAPI) 또는 Promise chain 활용
   - 스테이지별 진행 표시 (progress indicator)
4. easing, duration, delay 등 세밀한 조정 여부
5. 다중 CAT 활용: wpf-animation.pen의 4개+ 카테고리 기법 통합
```

### 축 W3: 독창적 확장 & 완성도 (30점)

디자인 참조를 넘어 **창의적 확장**과 **페이지 완성도**를 달성했는가.

| 점수 | 기준 |
|------|------|
| 0 | 단순 요소 나열, 페이지 완성도 미흡 |
| 12 | 기본 레이아웃 완성, 디자인 요소 단순 배치 |
| 22 | 반응형 + 인터랙티브 요소 + 일관된 테마/UX 흐름 |
| 26 | 디자인에 없던 창의적 인터랙션 추가 + 전체 UX 스토리라인 완성 + 접근성 고려 |
| 30 | 위 기준 + **내러티브 시퀀스 애니메이션** (AI Bloom 등) + **파라미터 조절 UI** + **외부 이미지 불필요(SVG/Base64 자체 생성)** |

**판정 방법:**
```
1. 반응형: @media 쿼리, 유동 레이아웃 존재 여부
2. 인터랙션: 클릭/스크롤/hover 이벤트 핸들러의 다양성
3. UX 스토리: 섹션 간 논리적 흐름
4. 접근성: semantic HTML, color contrast, keyboard navigation
5. 독창성: .pen에 없는 새로운 인터랙션 패턴
6. 내러티브 시퀀스: 스테이지별 진행되는 스토리 애니메이션 (예: Seed→Bloom)
7. 파라미터 제어: 슬라이더 등으로 애니메이션 파라미터 실시간 조절
8. 자체 리소스: 외부 이미지(PNG) 없이 SVG Blob URL, Base64, CSS 등으로 해결
```

## 7. 평가 절차

```
Step 1. 트리거 감지
  - Case A: WebSearch로 WPF 조사 + wpf-animation.pen에 새 카드 추가
  - Case B: wpf-animation.pen 참고 + 프로젝트 요구사항으로 새 .pen 디자인 완료
  - Case W: .pen 파일 참조 + HTML/CSS/JS 파일 생성 완료

Step 2. 컨텍스트 수집
  → Case A: wpf-animation.pen 기존 vs 신규 카드 비교
  → Case B: 사용자 요구사항 목록 + 정적 화면 + 애니메이션 가이드 확인
  → Case W: .pen 컴포넌트 → HTML 매핑

Step 3. 3축 채점
  → 각 축별 판정 방법에 따라 채점
  → 축별 근거(evidence) 1줄씩 기록

Step 4. 결과 출력
  → 총점 + 등급 + 축별 점수 + 매핑 테이블
  → recording 단계로 전달 (XP 계산용)
```
